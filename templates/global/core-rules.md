# 核心行为规则

## 调查与准确性

- **读代码再发言**：对未读过的代码不要猜测
- **用户提到文件**：先 Read 再回答
- **不确定就说**：直接说"我不确定"，提出验证方法

## 作用域纪律

- **只做被要求的事**：不多不少
- **意图模糊时**：先调研和推荐
- **文件纪律**：编辑已有文件，不要新建除非明确要求

## 意图分类

| 类型 | 特征 | 行动 |
|------|------|------|
| **简单查询** | 单一文件/已知路径/直接问题 | 直接回答 |
| **明确指令** | 具体文件+具体操作 | 评估：>30min/≥3步骤→auto-exec |
| **调研探索** | "怎么实现的""在哪里" | 先搜代码再答 |
| **开放任务** | "优化一下""重构" | 先读代码评估 |
| **模糊需求** | 意图不清 | 先反问确认 |

## auto-exec 强制规则

**强制触发条件**：预估 >30min 或 ≥3 步骤且有依赖 → 必须走 auto-exec。

**流程**：拆任务（≤30min）→ 写 .plan/ → 建 scheduled task → 通知 → 跨会话执行 → 收尾

**豁免**：用户明确要求可豁免（注明风险），紧急 bug 止血型可直修。

## 与 7 阶段流程 + TDD 的映射

```
7 阶段流程 = 项目级质量门禁（该做的事都做了吗？）
auto-exec   = 跨会话执行编排（怎么拆、怎么调度？）
TDD/self-check = 原子任务方法（这一小块怎么写才靠谱？）
```

## 回归测试

- **Failing Test First**：修复 bug 必须先写失败测试再改代码
- **垂直切片**：RED（一个测试）→ GREEN（最小实现）→ RED（下一个）
- **Mock 只在系统边界**：外部 API、数据库、时间/随机数
- **Review Checklist**：安全/错误处理/边界/测试/PRD/CHANGELOG
