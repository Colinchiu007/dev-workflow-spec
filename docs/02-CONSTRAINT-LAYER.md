# 约束层 — L1

## 架构

约束层由三层构成，每层的定位和加载时机不同：

```
.clinerules（全局）        ← 机器可读，每次代码生成前加载
├── FUSE 硬门禁（规则 #0）
├── auto-exec 强制门禁
├── 架构红线
├── 代码风格
├── 安全规范
├── 测试要求
├── 协作规则
└── 7 阶段流程门禁

.claude/rules/
├── core-rules.md           ← 人类可读，每次会话启动加载
│   ├── 调查与准确性
│   ├── 作用域纪律
│   ├── 通讯风格
│   ├── 意图分类 (Intent Gate)
│   ├── auto-exec 强制规则
│   ├── 与 7 阶段流程 + TDD 的映射
│   ├── 验证与安全
│   ├── Agent Loop 纪律 (Manus Event Stream)
│   └── 回归测试与审查清单
│
└── prompt-defense.md       ← 安全基线，每次会话启动加载
    ├── 角色锁定
    ├── 数据保护
    ├── 代码安全
    ├── 输入怀疑
    └── 外部数据隔离
```

## 设计原则

1. **冗余是可接受的**：`.clinerules` 和 `core-rules.md` 内容有重叠。机器读 .clinerules，人读 core-rules.md，两套互不替代
2. **机器可读优先**：能用规则表达的就写入 `.clinerules`，减少对 AI 理解能力的依赖
3. **门禁升级路径**：软规则（core-rules.md）→ 硬约束（.clinerules）→ Git 钩子（.githooks）→ CI 门禁
4. **紧急 bypass**：所有门禁都有明确的跳过条件和审批路径（如止血型 bug 不走 auto-exec）

## .clinerules 格式

使用 YAML 注释风格（`#` 开头），每条规则以 `- rule:` 开头：

```yaml
## 架构
- rule: "所有模块间数据交换必须使用 shared_models 中定义的 Pydantic v2 模型"
- rule: "domain 层不能 import repository 层"
```

## core-rules.md 格式

标准 Markdown，章节标题 + 列表/表格：

```markdown
## 调查与准确性

- **读代码再发言**：对未读过的代码不要猜测
- **用户提到文件**：先 Read 再回答
```

## Intent Gate 分类

每次收到请求时，先快速分类再决定行动路径：

| 类型 | 特征 | 行动 |
|------|------|------|
| 简单查询 | 单一文件/已知路径/直接问题 | 直接回答 |
| 明确指令 | 具体文件+具体操作 | 评估：>30min/≥3步骤→auto-exec |
| 调研探索 | "怎么实现的""在哪里" | 先搜代码再答 |
| 开放任务 | "优化一下""重构""加功能" | 先读代码评估 |
| 模糊需求 | 意图不清、范围不明 | 先反问确认 |
